# 测试策略制定 Skill

## 激活场景
- 需求分析完成后，开始制定整体测试方案
- 用户询问"应该测哪些方面？"
- 用户需要评估测试覆盖是否充分
- 项目有时间约束，需要优先级排序

## 测试类型选择决策树

### 🔹 功能测试（必选）
**适用条件**：所有功能性需求
```
正向流程测试   → 验证核心业务路径（Happy Path）
逆向流程测试   → 验证错误处理与异常场景
边界条件测试   → 验证输入边界处理
等价类测试     → 验证数据分类处理
```

### 🔹 接口测试（API项目必选）
**适用条件**：存在HTTP API、RPC、消息队列
```
契约验证       → 请求/响应格式符合接口文档
状态码验证     → HTTP状态码语义正确
认证鉴权       → Token/Session/OAuth验证
幂等性验证     → 重复请求处理
```

### 🔹 安全测试（涉及用户数据必选）
**适用条件**：认证、授权、用户数据、支付
```
SQL注入测试    → 数据库查询注入防护
XSS测试        → 跨站脚本防护
CSRF测试       → 跨站请求伪造防护
越权测试       → 水平/垂直越权验证
敏感数据保护   → 密码、身份证、手机号脱敏
```

### 🔹 性能测试（高并发场景必选）
**适用条件**：用户量>1000、高频接口、数据密集型操作
```
基准性能测试   → 单用户响应时间基线
负载测试       → 预期并发用户数下的表现
压力测试       → 极限并发用户数
稳定性测试     → 7×24小时长时间运行
```

### 🔹 兼容性测试（多端产品选择）
**适用条件**：Web端、移动端、多浏览器
```
浏览器兼容     → Chrome/Firefox/Safari/Edge
设备兼容       → iOS/Android 多机型
分辨率适配     → 375px/768px/1024px/1920px
OS版本兼容     → iOS 14+/Android 8+
```

### 🔹 可用性测试（ToC产品选择）
**适用条件**：面向终端用户的消费级产品
```
交互逻辑验证   → 操作反馈是否即时
错误提示质量   → 错误信息是否清晰可操作
无障碍验证     → WCAG 2.1 AA级标准
```

## 优先级策略矩阵

基于**业务影响 × 风险概率**进行优先级评定：

```
              | 低概率 | 高概率
高业务影响    |   P1   |   P0
低业务影响    |   P3   |   P2
```

### 优先级定义（强制遵循）
| 级别 | 名称 | 执行时机 | 通过率要求 | 典型场景 |
|------|------|---------|-----------|---------|
| **P0** | 阻塞级 | 每次构建/发布前 | 100% | 核心支付流程、用户登录 |
| **P1** | 高优先级 | 每个Sprint结束 | ≥95% | 主要业务功能、数据准确性 |
| **P2** | 中优先级 | 版本发布前 | ≥90% | 辅助功能、优化项 |
| **P3** | 低优先级 | 有时间则执行 | 尽力覆盖 | 边缘场景、UI细节 |

## 特殊策略场景

### 时间紧迫时的策略（冒烟测试套件）
> 当发布时间压缩时，快速构建最小有效测试集：
> 1. 只执行P0级用例（核心流程不超过20个用例）
> 2. 覆盖所有主要功能的正向流程
> 3. 覆盖最高风险的异常场景（支付失败、数据丢失）

### 新功能 vs 存量功能策略
- **新功能**：全面覆盖，包含边界和异常（设计深度=深）
- **存量功能**（受新功能影响）：聚焦影响区域（设计深度=中）
- **存量功能**（未受影响）：仅冒烟验证（设计深度=浅）
