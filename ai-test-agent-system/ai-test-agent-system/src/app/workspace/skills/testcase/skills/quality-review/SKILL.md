# 测试用例质量评审 Skill

## 激活场景
- 每批测试用例生成完成后（自动执行轻量自检）
- 用户要求"审查用例质量"或"评分"
- 用户要求生成测试质量报告
- 用例提交评审前的最终检查

---

## 质量评审维度（Quality Dimensions）

### 维度一：完整性检查（Completeness）- 权重30%

对照需求功能矩阵，逐项验证覆盖情况：

```
✅ 检查项：
□ 所有功能点（Feature）均已设计对应测试用例
□ 每个功能点有正向用例（Happy Path）
□ 每个功能点有至少1个异常/负向用例
□ 关键业务流程的端到端场景已覆盖
□ 所有输入字段已考虑边界值
□ 所有P0功能点用例数量 ≥ 3个
□ 安全相关功能（认证/授权/输入验证）已有安全测试用例
□ 状态机转换的所有路径已覆盖

评分规则：
满足检查项数 / 总检查项数 × 100 = 完整性得分
```

### 维度二：准确性检查（Accuracy）- 权重25%

逐条用例检查预期结果的可验证性：

```
✅ 检查项（针对每条用例）：
□ 预期结果是否具体可验证（禁止"正确显示"、"正常"等模糊词）
□ 测试步骤是否清晰无歧义（每步只有一个操作）
□ 测试数据是否具体（有实际值而非"有效数据"等描述）
□ 前置条件是否可重现（避免依赖"上一步的结果"）
□ 用例是否真正只验证一个检查点（原子性）

❌ 红线词汇（出现即扣分）：
- "正确"、"成功"、"失败"（需改为具体状态描述）
- "合适"、"适当"、"美观"（主观描述）
- "等待"、"稍后"（时间不明确）
- "相关"、"对应"、"正常"（逻辑不明确）
```

### 维度三：有效性检查（Validity）- 权重25%

评估测试用例的实际执行价值：

```
✅ 检查项：
□ 每条用例是否能发现至少一类缺陷（不是形式用例）
□ 设计方法是否与测试场景匹配
□ 测试数据是否能有效触发目标逻辑
□ 是否存在重复用例（逻辑完全相同只换了一个数据）
□ 高优先级测试点是否使用了更严格的测试方法
□ 安全测试数据是否具备真实攻击价值

有效性分级：
高价值用例：能直接发现核心功能缺陷（计分×1.5）
标准用例：  验证正常逻辑（计分×1.0）
低效用例：  与其他用例重复或价值低（计分×0.5）
冗余用例：  完全重复，建议删除（计分×0）
```

### 维度四：可执行性检查（Executability）- 权重20%

确保用例可以被测试人员独立执行：

```
✅ 检查项：
□ 前置条件是否可独立准备（不依赖其他用例的执行结果）
□ 测试步骤是否足够详细（新人可独立执行）
□ 测试数据是否完整提供（不需要测试人员自己查找）
□ 预期结果是否可客观判定（有Pass/Fail标准）
□ 用例是否与环境紧密耦合（最好能在多环境执行）
□ 是否说明了所需的账号权限
```

## 自检触发规则（轻量版）

每次生成完一个模块的用例后，自动执行快速自检：

```
快速自检项（10项）：
1. □ 模块所有功能点都有对应用例
2. □ 无一条用例使用模糊预期结果
3. □ 每个用例都有具体测试数据
4. □ P0用例数量合理（至少3条）
5. □ 包含至少1条安全相关用例（如有用户输入）
6. □ 包含至少1条异常场景用例
7. □ 用例编号无重复
8. □ 前置条件均可独立准备
9. □ 测试步骤步数合理（5-15步）
10. □ 预期结果包含UI、数据两个层面验证

通过9-10项 → 输出"✅ 自检通过"
通过7-8项  → 输出"⚠️ 自检发现小问题"并列出
通过<7项   → 输出"❌ 需要重新设计"并列出所有问题
```
